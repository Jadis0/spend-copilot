# spend-copilot

وكيل يحوّل إيصال مصروف فوضوي (صورة) إلى قيد نفقة منظّم، مصنّف، ومفحوص مقابل سياسة الشركة — مبني كنموذج مصغّر لمشكلة SiFi الحقيقية في إدارة النفقات المؤسسية.

## لماذا هذا المشروع

بُني ليطابق مباشرة متطلبات دور AI Engineer في SiFi (فنتك سعودية، ترخيص EMI من ساما): استخراج منظّم من مستندات مالية، تصنيف نفقات، فحص امتثال، RAG، وكلاء مستقلون، أطر تقييم، وواجهات backend.

## المعمار
صورة إيصال (PDF/JPG)
↓
[1] استخراج منظّم        GPT-4o Vision + Structured Outputs (strict mode)
↓
[2] تحقق حسابي مستقل     مجموع البنود = subtotal ؟  subtotal + tax = total ؟
↓
[3] تصنيف هجين           قاعدة على التاجر أولاً → LLM (مع سياق البنود) للغامض
↓
[4] فحص السياسة (RAG)    تقطيع دقيق (bullet-level) + ChromaDB + embeddings
↓
[5] وكيل القرار          قواعد حتمية للحالات الواضحة → LLM للمنطقة الرمادية
↓
[6] FastAPI              POST /process-receipt
+
[7] إطار تقييم           ground_truth.json + قياس 4 أبعاد مستقلة

## قرارات تصميمية، ولماذا

**مرحلتان لا استدعاء واحد.** الاستخراج (رؤية) منفصل عن الحكم (منطق أعمال). هذا يسمح بتحقق حسابي حتمي لا يعتمد على "فهم" النموذج.

**تحقق حسابي مستقل عن اللغة النموذجية.** `check_math` و`check_totals` عمليتا جمع Python بسيطتان، لا استدعاء LLM إضافي. اكتشفتا أخطاء استخراج حقيقية في كل تشغيل تقريباً — بما فيها خطأ لم يظهر في تحذير النموذج نفسه (بند سعر مفقود بقيمة 5 ريال في إيصال Saska's، اكتُشف فقط عبر مطابقة subtotal).

**قاعدة قبل LLM، في كل مرحلة — لا التصنيف فقط.** طُبّق نفس المبدأ في التصنيف (يوم 2) وفي قرار الوكيل (يوم 4): قواعد Python حتمية للحالات الواضحة، LLM محفوظ للمنطقة الرمادية فقط. في تشغيل تجريبي، 5/7 قرارات نهائية جاءت من قاعدة صريحة، لا LLM.

**RAG بتقطيع على مستوى الجملة (bullet-level)، لا القسم.** التقطيع الأولي (قسم كامل = قطعة) أخفى قواعد فرعية محددة داخل نص أطول. التقطيع الدقيق (كل bullet قطعة) حسّن الدقة، لكن لم يحلّ كل الحالات (انظر القيود أدناه).

**عتبة "مراجعة" بشرية مقصودة، لا عيب.** النظام لا يُطلب منه دقة 100%. يُطلب منه معرفة متى لا يثق بنفسه، ويصعّد القرار لإنسان بدل التخمين الواثق.

## قيود مكتشفة فعلياً (لا افتراضية)

هذي ليست قائمة نظرية — كل بند مُوثّق بتشغيل فعلي أثناء البناء.

**1. عدم الحتمية حتى مع `temperature=0`.**
نفس الصورة، نفس الكود، تشغيلات متتالية أنتجت فروقات حسابية مختلفة (4.53 → 6.73 → 10.53 ريال فرق) في نفس البند. `temperature=0` يقلل التذبذب في اختيار الكلمات، لكن لا يُلغي التذبذب في "قراءة" الصورة نفسها. **الأثر العملي:** أي نظام تقييم يعتمد على تشغيلة واحدة لكل حالة هش بطبيعته؛ تقييم موثوق يحتاج تشغيلات متعددة لكل عينة.

**2. embeddings تربط بالمفاهيم العامة بقوة، لكن تضعف مع أسماء علامات تجارية محددة.**
استعلام "beer alcohol" ربط بقوة (مسافة 1.32) ببند منع الكحول. استعلام بنفس المعنى لكن باسم منتج محدد ("Miller Lite") ربط بضعف شديد (مسافة 1.68) — لم يظهر بند الكحول حتى ضمن أفضل 3 نتائج. **الأثر العملي:** فئات عالية الخطورة قانونياً/امتثالياً (كحول، رشاوى، مقامرة) تحتاج طبقة قاعدة صريحة (قائمة كلمات/علامات معروفة) فوق RAG، لا الاعتماد على embeddings وحدها.

**3. الأخطاء تتراكم عبر المراحل بصمت.**
خطأ ربط RAG (البند 2 أعلاه) انتقل تلقائياً لقرار الوكيل، الذي قارن مبلغ إيصال بحد سياسة غير ذي علاقة (200 ريال لاجتماعات جماعية، بدلاً من فحص "ممنوع كلياً" لبند كحول لم يُربط أصلاً). الوكيل لا "يعرف" أن RAG أخطأ — يتصرف بثقة كاملة بناءً على مدخل خاطئ. اكتُشف فقط عبر مقارنة القرار النهائي بحقيقة مرجعية مستقلة (`eval.py`)، لا بفحص كل مرحلة بمعزل عن الأخرى.

**4. القاعدة (rule-based) في التصنيف تفشل مع أسماء علامات تجارية حقيقية.**
قاموس أولي بكلمات عامة ("restaurant", "cafe") طابق 1/6 مطاعم حقيقية فقط (Taco Bell, Golden Bowl Teriyaki, Saska's — لا شيء منها يحوي كلمة تدل على "مطعم" في اسمها التجاري). 5/6 احتاجت LLM. **الأثر العملي:** نسبة قاعدة/LLM الحقيقية على بيانات واقعية أقل بكثير من الافتراض الأولي؛ قاموس القواعد يحتاج توسعة مستمرة مبنية على مراجعة حالات LLM المتكررة.

## التقييم

`eval.py` يقيس 4 أبعاد مستقلة على عينة من `ground_truth.json` (حالياً 7 عينات، يُوسَّع لـ 50):
- دقة الاستخراج (تاجر، مبلغ، عملة)
- دقة التصنيف
- دقة التحقق الحسابي (هل اكتُشفت المشاكل المعروفة، دون إنذارات كاذبة)
- دقة قرار الوكيل مقابل قرار متوقع مبني على حكم بشري

تشغيل نموذجي: استخراج 7/7، تصنيف 7/7، تحقق حسابي 6-7/7 (متذبذب، انظر القيد 1)، قرار الوكيل 6/7 — الحالة الفاشلة موثّقة ومرتبطة مباشرة بالقيد 2.

## التشغيل

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install openai pydantic python-dotenv chromadb numpy fastapi uvicorn python-multipart

# .env: OPENAI_API_KEY=sk-...

uvicorn api:app --reload
# POST http://127.0.0.1:8000/process-receipt  (multipart file upload)

python eval.py   # تقييم دفعي مقابل ground_truth.json
```
